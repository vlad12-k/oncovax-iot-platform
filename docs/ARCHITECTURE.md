# Architecture

## 1) Architecture intent

OncoVax is an event-driven cold-storage monitoring platform baseline implemented as a production-style repository architecture.

The architecture is designed to demonstrate a hosted operational baseline for telemetry ingestion, threshold-based alerting, API/UI operations, and observability. It is intentionally scoped as a serious engineering baseline rather than a fully hardened production environment.

Scope boundaries for this architecture:

- Telemetry is generated by software simulators in this repository.
- The platform models monitoring and alert workflows, not hardware fleet lifecycle management.
- The repository does not claim clinical certification or fully completed production hardening.

## 2) Core service architecture

The current architecture is composed of the following services and infrastructure layers:

- **Simulator (`services/simulator/`)**
  - Publishes software-generated telemetry to MQTT topics.

- **MQTT broker (Mosquitto)**
  - Provides telemetry message transport between producers and consumers.

- **Worker (`services/worker/`)**
  - Subscribes to telemetry topics.
  - Validates payloads.
  - Persists telemetry and alert signals to InfluxDB.
  - Persists operational alert lifecycle/audit records to MongoDB.

- **InfluxDB**
  - Time-series persistence for telemetry and alert measurements.

- **MongoDB / Atlas-backed persistence (`MONGO_URI`)**
  - Operational persistence boundary for alert lifecycle and acknowledgement records.

- **API service (FastAPI, `services/api/`)**
  - Exposes health and operational endpoints.
  - Serves dashboard assets for operator workflows.
  - Reads and updates operational records in MongoDB.

- **Dashboard/UI (`services/web/`)**
  - Operator-facing interface for summary, alert visibility, and acknowledgement interactions.

- **Grafana (`grafana/`)**
  - Observability dashboards backed by InfluxDB.

- **nginx ingress (`infra/nginx/nginx.conf`)**
  - Production-like ingress boundary for HTTPS routing, basic-auth protection, and route-level exposure controls.

- **Orchestration adapter (`services/orchestration_adapter/`)**
  - Repository-implemented adapter for demo/runtime-control topic mapping via MQTT.
  - Present in dev and production-like compose topologies.

## 3) Data and control paths

### Telemetry ingestion path

1. Simulator publishes telemetry to MQTT.
2. Worker subscribes to telemetry topics from Mosquitto.
3. Worker validates payloads and writes telemetry points to InfluxDB.

### Alert generation path

1. Worker evaluates telemetry against threshold logic.
2. Worker writes alert-series signals to InfluxDB.
3. Worker writes alert lifecycle/audit records to MongoDB.

### Operational API path

1. API reads operational records from MongoDB.
2. API serves summary/list/detail workflows.
3. API processes alert acknowledgement updates in MongoDB.
4. Dashboard/UI consumes API routes from the same service boundary.

### Observability path

1. Grafana queries InfluxDB for telemetry and alert visualization.
2. In production-like topology, Grafana is reached through protected nginx ingress routing.

### Runtime/demo-control path (implemented)

- Orchestration adapter consumes demo-control topics and publishes simulator runtime-control events over MQTT.
- This path is additive and does not replace the canonical telemetry ingestion authority (`MQTT -> worker`).

## 4) Trust and exposure boundaries

### Public-safe route boundary

- nginx exposes `GET /public-health` as a public-safe liveness route without basic auth.
- This route is intentionally narrow and should remain constrained by deployment perimeter controls.

### Protected operational surfaces

- Operational API/dashboard routes are behind nginx basic-auth in production-like ingress.
- Alert acknowledgement route patterns are separately protected and write-rate limited.

### Grafana boundary

- Grafana is not an anonymous public surface in production-like topology.
- nginx applies basic-auth protection for Grafana host routing.

### Ingress-layer vs application-layer controls

- Current protection is significantly ingress-layer driven (TLS scaffolding, route segmentation, basic auth, selected rate limiting).
- Application-layer authentication/authorization is not yet fully implemented as first-class API controls.

## 5) Persistence boundaries

### InfluxDB responsibility

- Stores telemetry measurements.
- Stores alert-series measurements for observability and trend analysis.
- Serves as Grafana query backend.

### MongoDB responsibility

- Stores durable operational alert lifecycle and acknowledgement records.
- Serves as the API’s operational persistence source for summary/list/detail/acknowledgement behavior.

### Atlas-backed persistence in hosted baseline

- Hosted baseline architecture supports MongoDB Atlas-backed persistence through `MONGO_URI` configuration.
- This preserves the same operational persistence contract while moving storage to managed infrastructure.

## 6) Environment architecture differences

### Local/dev (`infra/docker-compose.dev.yml`)

- Includes full local stack: simulator, Mosquitto, worker, API/UI, InfluxDB, MongoDB, Grafana, Node-RED, and orchestration adapter.
- Exposes development ports directly for local testing and iteration.

### Hosted baseline (`infra/docker-compose.yml`)

- Core hosted stack for Mosquitto, worker, API, InfluxDB, and MongoDB.
- Uses environment-file configuration and supports Atlas-backed persistence via `MONGO_URI`.

### Production-like ingress path (`infra/docker-compose.prod.yml` + nginx)

- Adds nginx reverse proxy and TLS certificate mounting.
- Uses protected ingress routing for operational surfaces.
- Exposes public-safe health path and authenticated operational paths.
- Includes simulator, orchestration adapter, and Grafana with production-style restart behavior.

Environment behavior and risk posture differ materially across these deployment modes and must be validated per target environment.

## 7) Out-of-scope / non-claims

This architecture intentionally does not claim the following:

- Physical medical device fleet integration as represented by this repository runtime.
- Certified clinical or regulated infrastructure status.
- Fully completed production hardening.

Current architecture should be treated as a production-style hosted baseline with implemented controls and documented operational/security guidance, while recognizing additional hardening is required before full release-grade production treatment.
