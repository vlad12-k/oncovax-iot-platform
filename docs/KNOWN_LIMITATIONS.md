# Known Limitations

This document describes current, material limitations of the OncoVax IoT Platform baseline as implemented in this repository. It is intended for release-facing engineering communication and should be read together with `README.md`, `SECURITY.md`, and current infrastructure configuration.

## 1) Scope limitations

- Telemetry in this repository is generated by software simulators.
- The repository does not include a physical device fleet, hardware provisioning layer, or field-device lifecycle management.
- This platform baseline is not represented as a certified clinical or regulated medical deployment.
- The repository demonstrates a production-style architecture and hosted baseline patterns, but scope remains limited to the implemented monitoring workflow and supporting operational surfaces.

## 2) Security limitations

- Current security controls are baseline controls, not a complete hardening program.
- Ingress protections exist in the production-like path (TLS scaffolding, route protection, and rate limiting on selected routes), but application-layer authentication and RBAC remain incomplete.
- Operational API and dashboard protection currently rely significantly on ingress policy and deployment configuration quality.
- Secret handling is practical (environment variables and env files) but is not equivalent to mature centralized secrets-management systems.
- Public-safe ingress behavior is intentionally narrow (`/public-health`), and this route should remain constrained by deployment context and perimeter controls.

## 3) Deployment limitations

- A hosted baseline deployment is implemented, but hardening required for fully release-grade production operations is still incomplete.
- Security and reliability outcomes depend on operator-managed configuration quality (TLS certificates, credentials, firewall scope, domain configuration, and exposure boundaries).
- Environment behavior differs materially across:
  - local/dev compose,
  - hosted baseline compose,
  - production-like ingress compose with nginx.
- Assumptions valid in one environment do not automatically hold in another; deployment controls must be validated per target environment.

## 4) Observability limitations

- Observability is implemented through InfluxDB and Grafana, with dashboard assets and operational visibility paths available.
- Monitoring coverage and dashboard maturity are still evolving and should not be interpreted as exhaustive production observability.
- Alert and telemetry visibility are sufficient for baseline operational workflows, but not a substitute for a full enterprise monitoring and incident-detection program.

## 5) Operational limitations

- Runbook and recovery/rollback guidance are present, but operational safety still depends on disciplined execution by operators.
- The platform is not a zero-touch autonomous operations environment.
- Service health, credential hygiene, ingress policy, and rollback execution require active human ownership.
- Recovery quality depends on maintaining reliable deployment practices, backups, and verification routines in each environment.

## 6) Project maturity statement

The project demonstrates a serious production-style architecture with a working hosted baseline, protected operational surfaces, and documented operational/security controls.

At the same time, it should not yet be treated as fully release-grade production infrastructure without additional hardening work, stronger application-layer security controls, and environment-specific operational assurance.
