# Security Policy

## 1) Security scope

This repository represents a production-style IoT monitoring baseline for cold-storage telemetry workflows.
Production-style in this context means architecture and operational patterns that mirror production topology, without claiming complete production hardening.
It includes MQTT ingestion, worker-side processing, FastAPI operational APIs, dashboard delivery, time-series storage, audit persistence, and ingress controls.

Telemetry in this repository is generated by software simulators. The repository does not represent a fleet of physical medical devices and should not be treated as certified clinical infrastructure.

Security controls in this project are baseline and evolving. Current artifacts demonstrate practical controls and operational patterns, but do not claim fully completed production hardening.

## 2) Environment boundary

Security properties differ across deployment modes:

- **Local/dev stack (`infra/docker-compose.dev.yml`)**
  - Intended for development and testing.
  - Includes additional dev/demo surfaces (for example Node-RED) not required for canonical ingestion correctness.
  - Uses local service exposure patterns that are not equivalent to hardened internet-facing deployment.

- **Hosted baseline (`infra/docker-compose.yml`)**
  - Core hosted reference stack for MQTT, worker, API, InfluxDB, and MongoDB.
  - Supports Atlas-backed operational persistence through `MONGO_URI`.
  - Represents an operational baseline, not a final hardened production posture.

- **Production-like ingress path (`infra/docker-compose.prod.yml` + `infra/nginx/nginx.conf`)**
  - Adds nginx reverse proxy, TLS certificate mounting, and route-level protection behavior.
  - Intended as production-like scaffolding and operational reference.
  - Still requires environment-specific hardening decisions before internet-facing production use.

## 3) Ingress and exposure model

Ingress behavior is defined by nginx route policy in `infra/nginx/nginx.conf`.
When nginx is not used (for example, direct local development access), equivalent ingress protections and route gating are not automatically provided by this policy file and must be handled by deployment context controls.

- **Public-safe boundary**
  - `GET /public-health` is configured as a public-safe route without basic auth.
  - This route is intended for limited liveness/health-style checks.
  - In current nginx config, this exact route does not have an explicit `limit_req` directive; monitor and constrain upstream exposure accordingly.

- **Protected operational surfaces**
  - `location /` and alert acknowledgement route patterns are basic-auth protected.
  - Grafana host routing is also basic-auth protected in current nginx configuration.
  - Operational APIs and dashboards are not intended to be anonymous public surfaces.

## 4) Current baseline controls

The repository currently represents the following baseline controls and practices:

- **nginx ingress protections**
  - HTTPS redirect on port 80 for configured domains.
  - Security headers (`X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `X-XSS-Protection`) in nginx config.
  - Route-level auth segmentation between public-safe and protected paths.

- **HTTPS/TLS scaffolding**
  - TLS certificate/key paths are explicitly configured in nginx.
  - Production-like compose mounts `/etc/letsencrypt` for certificate use.
  - Deployment guide documents certificate provisioning workflow.

- **Basic auth / route protection**
  - Basic auth is required on protected API/dashboard routes and Grafana in current production-like wiring.
  - `.htpasswd` is mounted into nginx in `infra/docker-compose.prod.yml`.

- **Rate limiting**
  - nginx defines request limiting zones and applies limits on general/protected traffic and acknowledgement writes.

- **Environment-based secret handling**
  - Runtime-sensitive values are expected via environment variables and `.env` files.
  - Repository docs instruct using `.env.example` templates and avoiding committed real secrets.

- **Restart/recovery operational guidance**
  - Deployment, runbook, and recovery documents provide restart, rollback, and post-recovery validation steps.
  - Production-like compose uses `restart: always` for service resilience behavior.

- **Observability and audit controls**
  - InfluxDB and Grafana support operational observability.
  - MongoDB stores operational alert lifecycle/audit records.
  - Runbook/threat model document operational checks and known residual risks.

## 5) Known security gaps

Current limitations that should be treated as open security gaps include:

- **Application-layer auth and authorization**
  - API-level authentication and RBAC are not fully implemented as first-class application controls.
  - Current route protection relies heavily on ingress-layer controls.

- **Secret management maturity**
  - Environment-variable and env-file handling is practical but not equivalent to dedicated secrets-management systems.

- **Production hardening incompleteness**
  - Baseline controls exist, but full production hardening and formal security assurance are not complete.

- **Certification and compliance non-claims**
  - This project does not claim clinical certification, regulatory compliance certification, or equivalent formal approvals.

- **Environment configuration risk**
  - Security posture depends on correct operator configuration (domain, TLS certs, firewall scope, credentials, and route exposure).

## 6) Safe usage guidance

Use this repository as an operational baseline and engineering reference, not as pre-certified critical infrastructure.

- Keep telemetry sources simulated unless a separately governed hardware integration is designed and reviewed.
- Keep protected operational surfaces authenticated; do not expose dashboard, operational API routes, or Grafana as anonymous public endpoints.
- Use TLS for internet-facing ingress and validate certificate lifecycle and renewal procedures.
- Restrict network exposure to required ports and expected sources only.
- Rotate credentials and avoid committing real secrets into source control.
- Follow runbook and recovery guidance for service restart/rollback instead of ad-hoc hotfix operations.
- Do not treat this repository as certified clinical infrastructure.

## 7) Reporting a vulnerability

If you identify a potential security issue:

1. Do not post exploit details in a public issue.
2. Prepare a private report containing:
   - affected component(s) or file path(s)
   - reproduction steps
   - potential impact
   - suggested mitigation (if available)
3. Share the report through a private maintainer contact path when available.
4. If no private channel is available, open a minimal public issue requesting a private security contact method, and avoid disclosing sensitive exploit details until maintainers respond.

No formal vulnerability-response SLA is currently documented in this repository.

For all reports, include enough technical detail for reproducibility while minimizing exposure risk.
