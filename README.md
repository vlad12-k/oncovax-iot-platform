# OncoVax IoT Platform

OncoVax is an event-driven cold-storage monitoring platform that models telemetry ingestion, excursion alerting, operational APIs, and observability in a hosted baseline. The system is built as a production-style architecture (MQTT, worker processing, time-series storage, audit persistence, API, and ingress controls), while remaining explicit about scope: telemetry is generated by software simulators, not physical medical devices; this repository is not a certified clinical deployment; and additional hardening is required before treating it as fully release-grade production software.

## What this project is

- A repository for a production-style IoT monitoring stack for cold-storage telemetry workflows.
- A concrete implementation of an ingestion and alert pipeline:
  - simulator -> MQTT -> worker -> InfluxDB + MongoDB -> FastAPI/dashboard -> Grafana.
- A hosted-baseline deployment model with documented ingress, health checks, operational runbook steps, and smoke validation.
- An engineering artifact focused on traceable behavior and operational transparency.

## What this project is not

- Not a physical medical device platform.
- Not a live fleet of hardware sensors; telemetry in this repository is simulated.
- Not a certified clinical or regulated medical deployment.
- Not an anonymous public dashboard deployment; protected operational surfaces are intentionally access-controlled.
- Not a claim of fully complete production hardening.

## Key capabilities

- MQTT telemetry ingestion and worker-side schema validation.
- Threshold-based excursion detection with alert persistence.
- Time-series telemetry and alert signals in InfluxDB.
- Durable alert/audit workflow records in MongoDB.
- FastAPI endpoints for health, summary, alert listing/detail, and acknowledgement.
- Lightweight operator dashboard served by the API service.
- Grafana dashboards for telemetry and alert trend visibility.
- Deployment variants for local development, hosted baseline, and production-like ingress scaffolding.

## Architecture overview

The platform uses a layered architecture:

- **Simulator** publishes software-generated telemetry.
- **MQTT broker (Mosquitto)** provides message transport.
- **Worker service** validates payloads, writes telemetry, and emits alert/audit records when thresholds are exceeded.
- **InfluxDB** stores time-series telemetry and alert measurements.
- **MongoDB (local or Atlas-backed via `MONGO_URI`)** stores operational alert lifecycle/audit records.
- **FastAPI service** exposes operational APIs and serves the web dashboard.
- **Grafana** provides observability views from InfluxDB.
- **nginx ingress (production-like compose path)** fronts hosted surfaces with HTTPS/basic-auth controls and route-level exposure boundaries.
- **Hosted server baseline** is documented for DigitalOcean-style deployment with external domain/TLS wiring.

## System components

- `services/simulator/` - telemetry simulator runtime.
- `services/worker/` - ingestion, validation, threshold evaluation, persistence logic.
- `services/api/` - FastAPI routes and API container.
- `services/web/` - operator dashboard frontend.
- `services/orchestration_adapter/` - demo/runtime-control topic adapter.
- `infra/docker-compose.dev.yml` - full local development stack.
- `infra/docker-compose.yml` - base hosted stack baseline.
- `infra/docker-compose.prod.yml` - production-like stack with nginx ingress.
- `infra/nginx/nginx.conf` - ingress routing, protected surfaces, and TLS references.
- `grafana/` - dashboard/export and provisioning artifacts.
- `schemas/` - telemetry and related contract artifacts.

## Data flow

1. Simulator publishes telemetry to MQTT topics.
2. Worker subscribes and validates payload structure.
3. Worker writes telemetry to InfluxDB.
4. Worker evaluates thresholds and writes alert signals to InfluxDB.
5. Worker writes alert lifecycle/audit records to MongoDB.
6. API reads operational records for summary/list/detail/acknowledgement workflows.
7. Dashboard and Grafana provide operator/observability views.

## Hosted baseline

The repository documents and supports a hosted baseline deployment model (including `oncovax.live` wiring in current nginx configuration) with production-style topology patterns.

Current hosted baseline characteristics:

- Domain/TLS ingress is handled through nginx configuration.
- Public-safe endpoint exposure is intentionally limited (for example, health-style public checks).
- Operational API/dashboard surfaces are protected rather than anonymously public.
- Atlas-backed operational persistence is supported through environment configuration.

This hosted baseline should be treated as an operational reference implementation, not as a final certified production environment.

## Security posture

Security controls in this repository are baseline and evolving.

What is currently represented:

- Ingress-level protection model with public-safe and protected route separation.
- HTTPS/TLS scaffolding and basic-auth protection in production-like nginx configuration.
- Rate-limiting directives in nginx configuration.
- Environment-variable-based secret configuration patterns.
- Threat model and runbook documentation for known risks and responses.

What this does **not** claim:

- Complete production hardening or formal security certification.
- Clinical/regulatory compliance certification.
- Full elimination of all operational risk.

## Operational proof points

The repository includes reproducible operational proof surfaces:

- **Live public-health route behavior** documented and smoke-testable.
- **Protected operational API behavior** validated through authenticated checks.
- **Alert pipeline evidence** from ingestion through persistence and API visibility.
- **Restart and recovery guidance** in runbook/recovery documentation.
- **Observability stack evidence** via Grafana dashboard artifacts and InfluxDB-backed panels.

These proof points demonstrate implementation behavior and operational patterns, while remaining within the project’s stated scope boundaries.

## Local development / verification

Canonical local verification path:

```bash
make verify-local
```

This path starts the dev compose stack and runs smoke checks.

Common checks:

```bash
curl -s http://localhost:8000/summary | python -m json.tool
curl -s "http://localhost:8000/alerts?limit=20" | python -m json.tool
```

Production-like smoke path (requires real hosted environment, domain, and credentials):

```bash
./scripts/smoke_test.sh --prod oncovax.live oncovax-operator '<password>'
```

## Known limitations

- Telemetry is simulated by software components in this repository.
- The repository does not include a physical device fleet.
- This is not a certified clinical or regulated medical deployment.
- Security and operational hardening are present as a baseline but remain incomplete for fully release-grade production treatment.

## Roadmap / next hardening priorities

- Add stronger application-layer authentication/authorization patterns.
- Tighten secrets management beyond environment-file conventions.
- Expand deployment hardening and exposure controls for hosted environments.
- Increase automated operational/security validation coverage.
- Continue improving observability and incident-response depth.
